GETLISTINGS API ENHANCEMENT - PAGINATION AND SORTING
=====================================================

OVERVIEW
--------
The getListings endpoint has been enhanced to support pagination and sorting while maintaining full backward compatibility with existing frontend implementations.

ENDPOINT
--------
GET /listings

QUERY PARAMETERS (All Optional)
-------------------------------
1. page (integer, default: 1)
   - Specifies which page of results to return
   - Only used when 'limit' is also specified
   - Example: page=2

2. limit (integer, default: 0)
   - Number of listings per page
   - When set to 0 or omitted, returns ALL listings (backward compatible)
   - When set to a positive number, enables pagination
   - Example: limit=10

3. sortBy (string, default: 'recent')
   - Determines the sorting order of listings
   - Case insensitive (recent, RECENT, Recent all work)
   - Available options:
     * 'recent' - Sort by newest first (created date descending)
     * 'older' - Sort by oldest first (created date ascending)
     * 'priceAsc' - Sort by price ascending (lowest to highest)
     * 'priceDesc' - Sort by price descending (highest to lowest)
   - Invalid values default to 'recent' sorting
   - Secondary sort criteria applied for consistency
   - Example: sortBy=priceAsc

RESPONSE FORMATS
----------------

1. WITHOUT PAGINATION (Backward Compatible):
   When no 'limit' is specified or limit=0, the response is exactly the same as before:
   
   Response: Array of listing objects
   [
     {
       "_id": "...",
       "title": "...",
       "description": "...",
       "price": 100,
       "createdBy": { "name": "John Doe" },
       ...
     },
     ...
   ]

2. WITH PAGINATION:
   When 'limit' is specified (> 0), the response includes pagination metadata:
   
   Response: Object with listings and pagination info
   {
     "listings": [
       {
         "_id": "...",
         "title": "...",
         "description": "...",
         "price": 100,
         "createdBy": { "name": "John Doe" },
         ...
       },
       ...
     ],
     "pagination": {
       "currentPage": 1,
       "totalPages": 5,
       "totalCount": 50,
       "hasNextPage": true,
       "hasPrevPage": false,
       "limit": 10
     },
     "sortBy": "recent"
   }

USAGE EXAMPLES
--------------

1. Legacy usage (still works):
   GET /listings
   - Returns all listings sorted by newest first
   - Response format unchanged

2. Basic pagination:
   GET /listings?page=1&limit=10
   - Returns first 10 listings with pagination metadata

3. Pagination with sorting:
   GET /listings?page=2&limit=10&sortBy=priceAsc
   - Returns second page of 10 listings sorted by price (lowest first)

4. Sorting only (no pagination):
   GET /listings?sortBy=priceDesc
   - Returns all listings sorted by price (highest first)
   - No pagination metadata in response

5. Advanced example:
   GET /listings?page=3&limit=5&sortBy=older
   - Returns 5 listings from page 3, sorted by oldest first

6. Case insensitive sorting:
   GET /listings?sortBy=PRICEASC
   - Works with uppercase/mixed case parameters

7. Invalid parameter handling:
   GET /listings?sortBy=invalidSort&page=0
   - sortBy defaults to 'recent', page validation returns error

8. Error examples:
   GET /listings?page=0          → 400 error: "Page number must be greater than 0"
   GET /listings?limit=-5        → 400 error: "Limit must be non-negative"
   GET /listings?sortBy=invalid  → Works fine, defaults to 'recent' sorting

FRONTEND IMPLEMENTATION NOTES
-----------------------------

1. BACKWARD COMPATIBILITY:
   - Existing frontend code will continue to work without any changes
   - The API automatically detects if pagination is requested based on the 'limit' parameter

2. PAGINATION IMPLEMENTATION:
   - Check if response has 'pagination' property to determine response format
   - Use pagination.hasNextPage and pagination.hasPrevPage for navigation buttons
   - Display pagination.currentPage / pagination.totalPages for page indicators

3. SORTING IMPLEMENTATION:
   - Provide dropdown/buttons for sorting options:
     * "Newest First" (sortBy=recent)
     * "Oldest First" (sortBy=older)
     * "Price: Low to High" (sortBy=priceAsc)
     * "Price: High to Low" (sortBy=priceDesc)

4. RECOMMENDED UI FLOW:
   - Add pagination controls (Previous/Next buttons, page numbers)
   - Add sorting dropdown/buttons
   - Add items-per-page selector (10, 20, 50 items)
   - When user changes sorting or items-per-page, reset to page 1
   - Validate user inputs before making API calls

5. ERROR HANDLING:
   - Handle 400 validation errors for invalid page/limit values
   - Display user-friendly error messages for validation failures
   - Invalid sortBy values are handled gracefully by the API (no errors)
   - Implement client-side validation to prevent invalid page/limit values

6. SORTING UI RECOMMENDATIONS:
   - Use case-insensitive parameter values (API handles both cases)
   - Provide clear labels for sorting options:
     * "Newest First" (recent)
     * "Oldest First" (older) 
     * "Price: Low to High" (priceAsc)
     * "Price: High to Low" (priceDesc)
   - Consider adding visual indicators for current sort direction

TECHNICAL DETAILS
-----------------
- All listings with state 'sold' are automatically filtered out
- Default sorting is by creation date (newest first) for consistency
- Pagination metadata is only included when limit > 0
- Total count is calculated efficiently using MongoDB's countDocuments
- The API uses skip() and limit() for efficient pagination

SORTING IMPROVEMENTS
--------------------
- Case insensitive sorting parameters (recent, RECENT, Recent all work)
- Secondary sort criteria for consistent ordering:
  * Date sorts use _id as tiebreaker for same timestamps
  * Price sorts use creation date as tiebreaker for same prices
- Input validation for page (must be > 0) and limit (must be >= 0)
- Graceful handling of invalid sortBy values (defaults to 'recent')
- Sort criteria mapping:
  * recent → { created: -1, _id: -1 }
  * older → { created: 1, _id: 1 }
  * priceAsc → { price: 1, created: -1 }
  * priceDesc → { price: -1, created: -1 }

MIGRATION STRATEGY
------------------
1. Phase 1: Deploy backend changes (already done)
2. Phase 2: Update frontend to use new features gradually
3. Phase 3: No breaking changes - old frontend code continues to work

This enhancement provides a smooth path for implementing modern pagination and sorting features while ensuring zero disruption to existing functionality.
